## What is this branch?

This branch contains the **pre-built static output** of the Orbiter landing page (`orbiter.sh`).

It is the compiled result of the Astro project at `apps/landing/` in the `main` branch. Every file here — HTML, CSS, JS, images — was generated by the build pipeline and committed automatically. There is no source code here, only deployable artifacts.

## Why a separate branch?

The `main` branch holds the full monorepo (CMS packages, CLI, integration, demo app, and the landing page source). Plesk needs a clean directory of static files to serve — no `node_modules`, no build tooling, no unrelated packages.

`deploy/landing` is that clean directory. Plesk is configured to sync this branch to the web root for `orbiter.sh`, so every successful build automatically goes live.

## Deployment flow

```
main (source)
  └── apps/landing/src/   ← edit here
        │
        │  push to main triggers:
        ▼
  GitHub Actions (.github/workflows/deploy-landing.yml)
        │
        │  npm run build --workspace=apps/landing
        │  → apps/landing/dist/
        ▼
  deploy/landing (this branch)
        │
        │  Plesk pulls / syncs this branch
        ▼
  orbiter.sh  ✓
```

The workflow triggers automatically on any push to `main` that touches `apps/landing/**` or the workflow file itself. It can also be triggered manually via **Actions → Run workflow** in GitHub.

## How to update the landing page

**Never edit files in this branch directly.** They will be overwritten on the next deploy.

Instead:

1. Edit source files in `main` under `apps/landing/src/`
2. Push to `main`
3. GitHub Actions builds and pushes the result here automatically
4. Plesk picks up the new files

```bash
# example
cd apps/landing
# edit src/pages/index.astro or src/styles/landing.css
git add .
git commit -m "feat: update landing page copy"
git push origin main
# → build runs → deploy/landing updated → orbiter.sh live
```
